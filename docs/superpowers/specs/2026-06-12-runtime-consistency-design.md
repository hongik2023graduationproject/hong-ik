# 런타임 일관성 (evaluator/VM 시맨틱 통일) 설계

Revision: v1 (2026-06-12)

**입력:** `2026-06-11-static-type-system-phase-b-design.md` 부록 B(불일치 6건)·부록 C(클래스 실측). 전 항목 2026-06-11 실측 기준.

**Goal:** 같은 `.hik` 코드가 evaluator(`--eval`)와 VM(기본)에서 **동일하게 동작**하도록 시맨틱을 통일한다. 통일 후 정적 타입 검사기의 "both-reject만 진단" 면제들을 해제해 진단 정밀도를 올린다. Phase C(strict 기본화)의 전제 조건.

**배경:** Phase A·B 실측에서 VM이 일관되게 더 관대함이 확인됨 (VM-only 거부 0건). 백엔드 선택이 프로그램 의미를 바꾸는 상태는 출시 불가 — 정식 사용자가 생기기 전인 지금이 시맨틱 확정의 적기다.

## 결정 테이블 (2026-06-12 사용자 확정)

|---|---|---|---|---|

| 3a | 이상 조합 산술/비교 (`1.5 + "a"`, `논리 < 논리` 등) | **거부** | VM | 부록 A.1 eval 거부 조합으로 통일 |
| 3b | `문자 < 문자` 등 문자쌍 비교 4종 | **허용** — 사전순, 결과 논리 | evaluator | 사용자 확정 (유용 기능 유지) |

## D1. 공통 선언 호환 규칙

#1과 #6은 같은 검사의 양면이다. 선언(`타입 이름 = 값`) 시 양 백엔드가 동일하게 적용할 규칙:

- 기본형: 선언 타입과 값 타입 **정확 일치** (정수↔실수 승격 없음)
- `타입?` (Optional): 해당 타입 또는 `없음` 허용
- 클래스 타입: 값이 해당 클래스 또는 **자손 클래스** 인스턴스면 허용 (부모 체인 탐색)
- 에러 메시지: evaluator의 기존 문구(`[줄 N] 선언에서 자료형과 값의 타입이 일치하지 않습니다.`)로 통일

VM은 현재 선언 타입 검사가 전무하다(컴파일 시 타입 토큰을 버림) — `정수 x = "a"`도 통과할 것으로 추정되며 plan Task 0에서 전수 실측 후 검사를 도입한다. 구현 위치(컴파일 시 타입 정보 동봉 + 런타임 검사 vs 컴파일 타임 거부)는 plan에서 결정하되, **재대입(`x = 값`)의 기존 동작 변화는 본 spec 범위에서 제외** — 선언만 다룬다 (재대입 검사 도입 여부는 실측 후 별도 판단).

## D2. 블록 스코프 (evaluator 수정)

- if(consequence/then)·while·foreach·forrange·try/catch 블록 본문은 **새 Environment**로 평가 (외부 참조 가능, 내부 선언은 블록 종료 시 소멸).
- VM의 기존 동작과 일치시키는 것이 목표 — VM은 수정하지 않는다.
- **주의:** 기존 evaluator 테스트·골든 중 누수에 의존하는 케이스가 있으면 깨진다 → plan Task 0에서 전수 확인 후, 누수 의존 코드는 의도 갱신(테스트 수정)으로 처리.
- 함수 본문·클래스는 기존 스코프 규칙 유지 (이미 양 백엔드 일치).

## D3. 이항 연산자 (VM에 거부 추가, evaluator에 문자열 비교 추가)

부록 A.1의 eval 거부 규칙을 **목표 시맨틱**으로 승격하되 문자쌍 비교만 예외:

- 산술(`+ - * / % ** & |`): 숫자쌍(+는 문자쌍 추가) 외 전부 거부 — **VM에 검사 추가** (eval-only-reject 94조합 중 산술·비교분)
- 비교(`< > <= >=`): 숫자쌍 + **문자쌍(사전순)** 허용, 그 외 거부 — VM은 이상 조합 거부 추가, **evaluator는 문자쌍 비교 구현 추가** (VM과 동일 결과: byte-wise 사전순 — VM 기존 구현 기준)
- 등호(`== !=`): 숫자쌍·문자쌍·논리쌍·한쪽 없음 허용, 그 외 거부 — VM에 거부 추가 (`정수 == 문자`는 양쪽 모두 이미 거부 — 현행 유지 확인만)
- 에러 메시지: evaluator 기존 문구(`좌항과 우항의 타입을 연산할 수 없습니다.`) 계열로 통일

## D4. 논리 연산자 (VM 수정)

- `&&`/`||`: 좌항 평가 → 논리 아니면 에러. 단락 평가 유지 (`false && X`, `true || X`는 우항 미평가). 우항이 평가되면 논리 요구, 아니면 에러. 결과는 항상 논리.
- VM의 "우항 값 그대로 반환"(JS식) 제거. evaluator의 기존 검사 시맨틱과 일치.

## D5. foreach (VM 수정)

- iterable이 배열·문자 외 타입(정수·실수·논리·사전·없음)이면 RuntimeException — 침묵 0회 순회 제거. (사전 순회는 양 백엔드 모두 미지원으로 통일 — `키목록()` 우회 안내. Phase E에서 키 순회 도입 검토.)
- 문자 iterable에서 원소 타입 표기가 `문자`가 아니면 에러 (eval 기존 동작) — VM의 elementType 무시 제거. 배열 iterable의 원소 타입 검사는 기존대로 양 백엔드 동작 유지 (plan Task 0에서 현행 실측 후 박제).
- 에러 메시지는 evaluator 기존 문구 재사용.

## D6. 검증 체계 — 패리티 픽스처 선행 (접근 A안)

각 이슈는 다음 순서로 진행한다 (이슈 단위 커밋):

1. 목표 동작을 단언하는 **골든 패리티 픽스처** 추가 — 기존 `PhaseZeroGoldens` 인프라가 eval·VM 양쪽에서 실행하므로, 픽스처 추가만으로 양 백엔드 동일 출력이 강제된다. 에러 케이스는 `시도/실패`로 감싸 출력 비교.
2. red 확인 (어긋난 백엔드에서 실패)
3. 해당 백엔드 수정 → both green
4. 전체 ctest + 기존 골든 무회귀

**기각/비포함 대안:** 빅뱅 수정(회귀 격리 불가), 공유 시맨틱 레이어(Object/VMValue 표현 차이로 추상화 비용 과대 — 단, 이항 연산자 허용 규칙의 진리표를 공용 inline 헤더로 추출하는 절충은 plan에서 검토).

## D7. 타입 체커 면제 해제 (마지막 단계)

런타임 통일 후, both-reject 원칙에 따라 체커를 확대한다:

| 항목 | 변경 |
|---|---|
| `&&`/`||` 우항 | 값 의존이 사라지므로 **우항도 TC601 검사** (논리 외 거부) |
| TC302 (foreach 순회 불가) | **부활** — iterable이 배열/문자/Any/Never 외면 진단 |
| forrange 실수 경계 | TC301 대상에 **실수 추가** (통일 후 both-reject) |
| 문자쌍 비교 | 면제 유지가 아니라 **정식 허용** — 결과 논리로 정밀 추론 (현재 Any) |
| 서브타입 대입 | 이미 허용 (Phase B-2) — 런타임과 완전 일치 확인만 |
| 부록 B 이슈 목록 | #1~#6 해소 처리, Phase B spec 갱신 |

## 호환성·공지

- **evaluator/vm 수정 가드를 본 작업에 한해 해제.** parser/lexer/ast/object(빌트인)은 여전히 무변경.
- WASM은 VM을 그대로 사용 — 웹 플레이그라운드 동작도 함께 바뀐다. CHANGELOG에 **변경 (Changed/Breaking)** 섹션으로 명시: 선언 타입 검사 도입(#1), 논리 연산자 값 반환 제거(#4), foreach 에러화(#5), 블록 스코프(#2)는 `--eval` 사용자에게만 해당.
- 새 RuntimeException들은 기존 `시도/실패`로 잡힌다 (양 백엔드 예외 처리 인프라 기존 동작).
