# VM 최적화 사이클 1 결과 — 2026-07-10

빌트인·컬렉션 경계 축소(D1~D4) + peephole 융합 패스(D5) 적용 결과. 스펙: `docs/superpowers/specs/2026-07-10-vm-optimization-cycle1-design.md`.

## 측정 방법론 — same-day 대조 (중요)

2026-07-08 베이스라인 JSON과 오늘 수치를 직접 비교하면 안 된다: **동일 베이스라인 커밋(e287422)을 오늘 재빌드·재측정하니 전 시나리오가 이틀 전 기록보다 5~18% 느리게 나왔다** (전원/열 상태 등 환경 드리프트, hashmap_ops 24.5→29.0ms 등). 따라서 이 문서의 개선율은 **같은 세션에서 순차 측정한 두 바이너리** 비교다:

- **대조(control)**: 베이스라인 커밋 e287422, worktree에서 MSVC Release 빌드 → `2026-07-10-baseline-control.json`
- **본 측정**: 사이클 1 완료 커밋 a192235, MSVC Release → `2026-07-10-cycle1.json`
- 각 5회 반복, **중앙값** 보고. CPython 3.14.2 레퍼런스도 오늘 재실행 (5회 중앙값).
- 방법론 유효성 검증: 이번 사이클이 건드리지 않은 트리워킹(eval/*)은 전부 ±2.4% 안 — 대조가 유효하다는 뜻.
- `tools/compare.py`는 이 머신의 numpy(MINGW 빌드)가 세그폴트해 사용 불가 — 동일 JSON을 직접 파싱한 중앙값 비교로 대체 (스크립트 산출값, 동일 정보).

측정 환경: CPU 16코어 4700MHz, Windows 11 Home, MSVC (VS 2022) Release.

VM = 트리워킹 대비 **8.8~24.8x** (class_method 8.8x ~ fib_recursive 24.8x). VM만 빨라졌으므로 격차는 전 시나리오에서 벌어졌다.

## 적용된 최적화 요약

1. **D1** `Builtin::function` by-value → const& — 호출당 인자 vector 복사 제거 (9850113)
2. **D2** OP_CALL/OP_INVOKE 빌트인 경로: dynamic_cast 1회화, 인자 버퍼 재사용, pop 루프 → resize (b6de4f0)
3. **D3** `길이` 네이티브 fast path — String/Array/HashMap 직접 계산, 그 외 제네릭 폴스루 (b6de4f0)
4. **D4** OP_INDEX_GET/SET 스칼라 인덱스 toObject 힙 할당 제거 (903507e)
5. **D5** peephole 융합 패스 — `SET_{LOCAL,GLOBAL,UPVALUE,MEMBER}+POP` → `*_POP` 4종, 점프 타깃 가드 + 오프셋·lines 재조립, 중첩 함수/클래스 재귀 (a192235)

## 다음 최적화 후보 (갱신)

1. **전역 변수 인라인 캐시** — arith_loop(268ms)의 지배 비용은 `OP_GET/SET_GLOBAL`의 매 접근 `unordered_map<string>` 해시 조회 (이번 사이클 탐색에서 확인). 최대 잠재 효과. rebind/erase 불변식 검토 필요.
2. **디스패치 루프 타이트닝** — frame/ip 로컬 캐싱. fib(콜 프레임)·전 시나리오 공통 이득, 전 핸들러 침습.
3. **컬렉션 원소 VMValue 내부 표현** — array/hashmap의 원소 저장이 여전히 `shared_ptr<Object>` (D4가 인덱스만 제거, 값 저장 할당은 잔존). Layer 3급 침습, 별도 스펙.
