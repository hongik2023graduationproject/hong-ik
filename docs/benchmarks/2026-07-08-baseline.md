# 벤치마크 베이스라인 — 2026-07-08

VM 최적화 Layer 1(싱글턴·정수 풀·UTF-8 캐시) + Layer 3(VMValue tagged union) 적용 상태의 첫 공식 베이스라인. 이후 최적화는 이 수치 대비로 평가한다 (`compare.py`, 원본 JSON: `2026-07-08-baseline.json`).

## 측정 환경

- CPU 16코어 4700MHz, Windows 11 Home
- hong-ik: MSVC (VS 2022) Release, `cmake-build-release`, 커밋 9326638
- 실행: `HongIkBenchmarks --benchmark_repetitions=5 --benchmark_format=json`, **중앙값** 보고
- CPython 3.14.2: `benchmarks/reference/*.py` 각 5회, 중앙값 (프로세스 시작 시간 제외 — 스크립트 내부 perf_counter)

## 해석

**바이트코드 VM은 트리워킹 대비 전 시나리오 9~25배.** 함수 호출이 지배하는 fib_recursive에서 격차가 최대(24.7x) — VM의 콜 프레임이 트리워킹의 환경 체인 생성보다 압도적으로 싸다. 이 비율이 쇼케이스의 핵심 수치다.

**CPython 대비 9~17배 느림.** 수십 년 최적화된 CPython과 교육용 언어의 격차로는 건전한 밴드이고, 주목할 것은 분포다: 스칼라 연산(arith 9.8x, fib 9.1x)은 상대적으로 선방하는 반면 **컬렉션·빌트인 경계(array 15.1x, builtin 16.8x, hashmap 13.4x)가 뒤처진다** — Layer 3가 스칼라를 VMValue로 인라인화했지만 컬렉션과 빌트인 호출은 여전히 `shared_ptr<Object>` 힙 경로 + 경계 변환을 타기 때문. 다음 최적화가 노릴 지점이 여기다.

**프론트엔드는 문제 없음.** 약 1,900줄 소스 기준 파스 2.6ms + 컴파일 0.9ms — LSP 실시간 재분석(didChange마다 전체 재파싱)에 충분한 여유.

측정 축 주의사항(Task 2 리뷰 관찰): builtin_calls는 루프 제어 비중이 대략 절반이라 순수 빌트인 오버헤드의 하한으로 읽을 것. class_method는 할당+디스패치 혼재. string_concat은 O(n²) 특성상 크기 변경 시 비선형.

## 다음 최적화 후보 (데이터 근거)

1. **빌트인 경계 변환 축소** — VM/CPython 최악(16.8x). `OP_CALL`/`OP_INVOKE` 빌트인 경로가 매 호출 `vector<shared_ptr<Object>>`를 만들고 VMValue↔Object 변환. 인자 버퍼 재사용 또는 고빈도 빌트인(길이 등)의 VMValue 네이티브 경로.
2. **Layer 2 잔여: peephole 패스 구현** — `Compiler::optimize()`가 placeholder. `OP_SET_LOCAL n; OP_POP; OP_GET_LOCAL n` 병합, no-op 점프 제거 등은 arith_loop(루프 본문 대입 반복)에 직접 효과. 저위험.
3. **컬렉션 연산 힙 경로** — array/hashmap의 원소 접근·삽입이 Object 할당 유발. 원소를 VMValue로 보관하는 내부 표현 전환은 효과 크지만 Layer 3급 침습 — 별도 스펙 필요.
4. (관찰) 트리워킹 개선은 우선순위 아님 — 기본 백엔드가 VM이고, 격차 자체가 쇼케이스 수치.
